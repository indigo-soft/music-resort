# Пропозиції щодо покращення коду й продуктивності

Документ містить дві частини:

- Частина A — Пропозиції щодо покращення коду (архітектура, якість, продуктивність, тестування, DX).
- Частина B — Пропозиції щодо підвищення продуктивності команди (процеси, інструменти, комунікація, CI/CD).

---

## Частина A. Покращення коду

### A1. Архітектура і розподіл відповідальностей

1) Розділити «оркестрацію» і «бізнес-логіку»:
    - Команди (src\Command\*) вже працюють як thin-controllers — це добре. Зберігати фокус команд на: парсинг
      аргументів, виклик сервісів, репортинг.
    - Посилити інкапсуляцію в сервісах (src\Service\Mp3ResortService, Mp3DeduplicateService, Mp3CleanService):
        - Повернення результатів через невеликі DTO/Result-обʼєкти замість «масивів зі статусами». Напр.,
          Result{int $status, int $processed, int $errors}.
        - Уніфікувати сигнатури методів: resort(), deduplicate(), clean() — однакова структура результатів.

2) Ініціалізація конфігурації:
    - Переконатися, що ConfigService::init() викликається в «ранньому бутстрапі» (наприклад, у bin\console) до виконання
      команд. Це унеможливить випадки, коли ConfigService::get('app.debug') недоступний.

3) Узгодженість понять «MP3» vs «Audio»:
    - В описі згадується mp3/flac/m4a. Для ясності або перейменувати сервіси з Mp3* на Audio*, або уніфікувати підтримку
      в сервісах, що існують, і явніше описати це в PHPDoc/README.

4) Повторне використання логіки:
    - Нормалізація артистів і ключів зустрічається в різних сервісах (напр., MusicResort\Service\Mp3ResortService::
      extractFirstArtist, MusicResort\Service\Mp3DeduplicateService::normalizeKey). Варто винести у спільний хелпер (
      напр., ArtistUtils) або до src\Helpers\helpers.php з окремими функціями, щоб забезпечити єдину поведінку.

5) Суворі типи та значення за замовчуванням:
    - Уже використовується declare(strict_types=1) — зберігати.
    - Додати явні типи скрізь, де можливо (null/nullable), уникати «змішаних» масивів у публічних API.

### A2. Якість коду і стиль

1) Статичний аналіз:
    - Додати PHPStan (рівень 7–8) з базовою конфігурацією; інтегрувати в CI та composer scripts.
2) Код-стайл:
    - Додати friendsofphp/php-cs-fixer (або PHP_CodeSniffer з PSR-12); налаштувати «format on CI» та pre-commit hook.
3) Перевірка абстракцій:
    - Використати pest-plugin-arch або ta-tikoma/phpunit-architecture-test (вже є в vendor) для фіксації архітектурних
      правил (наприклад, команди не використовують файлову систему напряму, лише сервіси).

### A3. Продуктивність і надійність виконання

1) Finder і фільтрація:
    - Вказувати чіткі маски розширень у Finder та виключення системних тек (напр., @eaDir, .AppleDouble) для зменшення
      IO.
    - Параметризувати максимальну глибину пошуку, якщо релевантно.
2) Дедублікація:
    - Використати двофазну стратегію: первинна група по (розмір файлу, довжина, CRC/MD5), а вже потім детальніша
      перевірка через getID3. Це зменшить кількість дорогих аналізів.
    - Кешування метаданих (в тимчасовому файлі або sqlite) між запусками для великих колекцій.
3) Dry-run семантика:
    - Усі операції з файловою системою загорнути в методи, які враховують dry-run (у вас це вже фактично так, але варто
      централізувати для однорідності та зручних тестів).
4) Обробка помилок:
    - Ввести чіткі класи винятків (напр., MusicMetadataException вже існує — розширити класифікацію:
      InvalidTagException, UnsupportedFormatException). Це полегшить повідомлення користувачу та retry-логіку.

### A4. Тестування

1) Unit/Integration тести (Pest):
    - Юніт-тести для нормалізації артистів/ключів (property-based: пробіли, регістри, суфікси «feat.», «ft.» та ін.).
    - Інтеграційні тести сервісів на тимчасових теках (sys_get_temp_dir), що перевіряють dry-run та реальні зміни ФС.
2) Контрактні тести для CLI:
    - Перевірка music:all — коли destination не заданий, крок resort пропускається (RunAllCommand::runResortIfNeeded).
3) Швидкі smoke-тести на семплах з папки samples\.

### A5. DX (developer experience)

1) Composer scripts:
    - scripts.test = "pest -d"
    - scripts.cs = "php-cs-fixer fix --diff --dry-run"
    - scripts.cs-fix = "php-cs-fixer fix"
    - scripts.stan = "phpstan analyse src --memory-limit=1G"
2) Шаблони: CONTRIBUTING.md, ISSUE_TEMPLATE, PR_TEMPLATE із чеклістами (линт/тести/стат. аналіз).
3) Документація:
    - Короткий DEV.md з інструкціями локального запуску, стандартами гілок та релізів.

### A6. Конкретні точкові покращення в коді

- MusicResort\Service\ConsoleCommandService::isDryRun — читає опцію + app.debug. Додати явний пріоритет CLI опції над
  debug, або окремий прапор DEBUG_FORCE_DRY_RUN, щоб уникнути плутанини в проді.
- MusicResort\Service\Mp3ResortService::extractFirstArtist — винести розділювачі та правила нормалізації в
  конфіг/хелпер, додати юніт-тести на крайні кейси (feat., &, /, коми; довгі імена; не-ASCII).
- MusicResort\Service\Mp3DeduplicateService::normalizeKey — синхронізувати з правилами в ResortService; додати перевірки
  на стоп-слова типу "original mix", "remaster" (за бажанням, під прапор).
- RunAllCommand — додати опцію "--continue-on-error" для довгих пайплайнів, щоб не зупинятись на одному збої (за
  замовчуванням збережіть нинішню поведінку).

## Частина B. Підвищення продуктивності команди

### B1. Процеси розробки

1) Working Agreement + Definition of Done:
    - Чіткий DoD: лінт пройдено, стат. аналіз без помилок, тести зелені, дока оновлена.
2) Trunk-based development з короткими гілками:
    - Малі PR ≤ 300 рядків коду; фічі за прапорами.
3) Код-ревʼю з чеклістами:
    - Якість коду, зрозумілість, покриття тестами, виробнича безпека.

### B2. CI/CD

1) Автоматичні перевірки на кожен PR: composer validate, cs, stan, pest.
2) Нічні jobʼи на великих семплах для регресій продуктивності.
3) Релізний pipeline: семантичне версіонування, changelog, GitHub Releases.

### B3. Автоматизація локальної роботи

1) Pre-commit hooks (captainhookphp/captainhook або lefthook): запуск cs, stan (швидкий режим), pest --min.
2) Makefile/justfile або Composer scripts як єдині команди для розробника.

### B4. Комунікація і знання

1) технічне демо 1× на 2 тижні; ретро 1× на 2–4 тижні з конкретними action items.
2) ADR (Architecture Decision Records) для ключових рішень.
3) Короткі навчальні нотатки («рецепти») в docs/recipes/ (напр., «як запускати прискорену дедублікацію на зовнішньому
   диску»).

### B5. Метрики та видимість

1) Engineering metrics: PR cycle time, lead time, flaky tests rate.
2) Проста телеметрія CLI (у сухому режимі) — анонімна статистика тривалості кроків, щоб бачити вузькі місця.

## Швидкі перемоги (Quick Wins)

- Додати PHPStan і CS Fixer + composer scripts.
- Винести нормалізацію артистів у хелпер і покрити тестами.
- Додати опцію --continue-on-error у music:all (за потреби).
- Налаштувати базовий CI (GitHub Actions): validate → cs → stan → pest.

## Довгострокові кроки

- Кеш метаданих і двофазна дедублікація для великих колекцій.
- Перейменування Mp3* у Audio* або чіткіший опис підтримуваних форматів.
- Введення DTO для результатів сервісів і поступова типізація публічних API.
